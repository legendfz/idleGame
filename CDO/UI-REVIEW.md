# UI-REVIEW.md — UI 一致性审查报告

**版本**：v1.0
**日期**：2026-03-01
**作者**：CDO
**审查范围**：CTO/idle-game/src/ vs CDO/UI-MOCKUPS.md + UI-DESIGN.md + UI-SPEC.md

---

## 总结

审查发现 **23 项差异**，其中 P0 关键 5 项，P1 重要 9 项，P2 建议 9 项。CTO 实现了核心框架和游戏循环，但 UI 层面在动画、交互细节、数字格式化等方面与 Mockup 存在偏差。

---

## P0 — 关键差异（影响核心体验）

### P0-1: 背包使用网格布局而非列表布局
- **Mockup**：背包采用列表行布局，每行显示 `{类型ico} {品质前缀}{名称} {属性} {标签}`
- **实现**：`BagView` 使用 `grid-template-columns: repeat(4, 1fr)` 网格布局，只显示 emoji + 名称 + 等级
- **影响**：丢失了物品属性速览、`[装备中]` 标签、品质前缀（○●◆★✦✧），玩家无法快速对比物品
- **建议**：改为单列列表，每项 3 行（名称/属性/标签），参照 EQUIPMENT-UI.md §1.2

### P0-2: 品质系统与设计不一致
- **Mockup**：品质 = 凡品○ / 良品● / 上品◆ / 极品★ / 仙品✦ / 神品✧（6 级）
- **实现**：品质 = common⬜ / spirit🟢 / immortal🔵 / divine🟣 / chaos🟡（5 级），用彩色方块而非 Unicode 前缀
- **影响**：品质视觉语言完全不同，emoji 方块在小字号下辨识度低于符号前缀
- **建议**：将 `QUALITY_INFO` 的 label 改为符号前缀体系，品质等级改为 6 级对齐设计规范

### P0-3: 缺少战斗飘字 / 伤害动画
- **Mockup**：定义了 7 种飘字（普通/暴击/技能/治疗/格挡/闪避/反伤），含 CSS @keyframes
- **实现**：所有伤害仅以战斗日志文字形式呈现，无任何飘字 DOM、无动画效果
- **影响**：战斗最核心的视觉反馈完全缺失，打击感为零
- **建议**：实现飘字组件，至少先做普通伤害和暴击飘字（参照 BATTLE-VFX.md §1）

### P0-4: 缺少离线收益中的装备掉落
- **Mockup**：离线弹窗显示 `🪙 / ✨ / 💊` 等多种收益
- **实现**：`OfflineReportModal` 仅显示灵石和经验，无装备/丹药/蟠桃离线掉落
- **影响**：离线收益体验单薄
- **建议**：补充离线装备掉落计算（至少模拟 boss 击杀次数 × 掉率）

### P0-5: 战斗界面缺少挂机收益统计区
- **Mockup**：战斗界面底部有独立的 `IdleStats` 区域：`💰 +350/秒  ✨ +42/秒 / ⏱️ 挂机中… hh:mm:ss`
- **实现**：无此区域，玩家不知道当前收益效率
- **建议**：在 `BattleView` 底部添加收益速率 + 挂机时间统计

## P1 — 重要差异（影响体验一致性）

### P1-1: 血条实现方式不同
- **Mockup**：使用 `█` / `░` 字符组成 10 格文字进度条
- **实现**：使用 CSS 填充条（`hp-bar-fill` div + width 百分比过渡）
- **评估**：CSS 实现视觉效果更好，**可接受**，但与"纯文字+符号"设计理念有偏差
- **建议**：保留 CSS 实现但考虑在日志内使用字符血条

### P1-2: 数字格式化不一致
- **Mockup**：使用中文单位（`1.2万`、`124万`）
- **实现**：使用英文单位（`1.2K`、`1.2M`、`1.2B`）
- **建议**：`formatNumber()` 改用中文单位：`<1万` 原值千分位，`≥1万` 用 `万`，`≥1亿` 用 `亿`

### P1-3: 装备详情弹窗缺少关键面板
- **Mockup**：详情弹窗含属性面板 + 强化预览 + 套装面板 + 属性对比
- **实现**：`EquipDetail` 仅显示基础属性 + 操作按钮，缺少：
  - 强化预览（`Lv.12→13, ⚡180→195 ▲15`）
  - 套装收集进度
  - 替换对比面板（`当前 vs 新装备`差值）
  - 装备描述/lore 文字
- **建议**：按 EQUIPMENT-UI.md §2 补全

### P1-4: 缺少强化动画
- **Mockup**：强化有 3 阶段动画序列（蓄力→锤击→结果），成功/失败不同表现
- **实现**：`enhanceItem` 直接修改数值，无视觉反馈
- **建议**：至少添加结果 Toast（成功绿框/失败红框）

### P1-5: 旅途界面缺少子关卡展开
- **Mockup**：当前章节展开显示子关卡列表（✅/🔥/🔒）+ 进度条
- **实现**：`JourneyView` 只显示章节列表，无展开交互，进度仅显示 `进度 n/N` 文字
- **建议**：点击当前章节展开子关卡，用树形 `├/└` 结构

### P1-6: BottomNav 缺少选中指示器
- **Mockup**：选中 Tab 下方有 2px 圆点指示器
- **实现**：仅 color 变为 accent，无圆点
- **建议**：添加 `::after` 伪元素圆点

### P1-7: 缺少 Tab 切换动画
- **Mockup**：Tab 切换时内容区 `opacity 0→1`，150ms ease
- **实现**：无过渡，瞬间切换
- **建议**：给 `.main-content` 加 `animation: fadeIn 150ms ease`

### P1-8: Boss 出场缺少特殊视觉
- **Mockup**：Boss 出场有 4 阶段序列（暗幕→弹入→打字台词→融入），含全屏遮罩 + scale 弹性动画
- **实现**：仅在日志中加一行 `═══ 🐉 BOSS: {name} ═══`
- **建议**：至少实现一个 Toast 弹窗版 Boss 出场（不需要完整 4 阶段，但需要视觉中断感）

### P1-9: 战斗日志暴击行缺少背景色
- **Mockup**：暴击日志整行微红底 `rgba(244,67,54,0.08)`，文字加粗
- **实现**：`.battle-log .crit` 仅改 color 为 accent（金色），无背景，无加粗
- **建议**：加 `background: rgba(244,67,54,0.08); font-weight: bold;`，颜色改为红色系

## P2 — 建议改进（打磨细节）

### P2-1: TopBar 高度不符规范
- **Mockup**：TopBar 56px + ResourceBar 32px = 88px 固定区
- **实现**：TopBar 使用 `padding: 8px 16px`，无固定高度，ResourceBar 合并在 TopBar 内
- **建议**：设定 `height: 56px` 或至少 `min-height`

### P2-2: 更多页面过于简陋
- **Mockup**：更多页 = 菜单列表（🏪商店/🏆成就/📊统计/⚙️设置），右侧 `→` 箭头
- **实现**：`SettingsView` 只有总游戏时间 + 存档/重置按钮
- **建议**：MVP 可简化，但至少保留菜单列表结构

### P2-3: 缺少物品获得弹窗
- **Mockup**：品质 ★ 以上装备获得时弹窗 + 闪光特效
- **实现**：仅日志一行文字
- **建议**：高品质（divine/chaos）装备掉落时弹一个 Toast

### P2-4: 队伍界面缺少队友切换
- **Mockup**：底部有 `[ 🐒悟空 ] 🧘唐僧 🐷八戒 🏔️沙僧` 切换栏
- **实现**：仅显示悟空，无队友系统
- **评估**：可能是 MVP 故意裁剪，非 bug。标记为 P2

### P2-5: 缺少空状态设计
- **Mockup**：背包空时显示 `📦 背包空空如也… 去战斗获取装备吧！`
- **实现**：空时显示 `暂无物品` 纯灰字
- **建议**：补充引导文案

### P2-6: 点击攻击区域过大
- **Mockup**：点击区高 ~80px，位于敌人面板和日志之间
- **实现**：点击区域 padding `var(--space-lg)` 即 24px，加上大号 emoji 和文字实际偏小
- **建议**：调整为 `padding: 20px 24px` 或设定 `min-height: 80px`

### P2-7: 战斗日志滚动方向
- **Mockup**：最新日志在顶部（新行从顶部推入）
- **实现**：`[entry, ...log]` 新条目 prepend 到数组 + `map` 正序渲染，效果一致 ✅
- **但**：CSS 无 `flex-direction: column-reverse` 或 scroll-to-top 逻辑，可能需要确认

### P2-8: 缺少卖出确认弹窗
- **Mockup**：卖出需要二次确认弹窗（双线框 + ⚠️ 提示）
- **实现**：`sellItem` 直接执行，无确认
- **建议**：添加 `confirm()` 或自定义确认弹窗

### P2-9: 强化无成功率机制
- **Mockup**：强化有成功率（显示 `████████░░ 80%`），失败不损坏装备
- **实现**：强化 100% 成功
- **评估**：MVP 简化可接受，但后续需加概率机制
