## 1. 纯前端 vs 需要后端？

**推荐：纯前端（PWA），无后端**

理由：
- 纯文字+符号，无需服务器渲染或实时通信
- Idle Game 的核心逻辑（离线收益计算）完全可以在客户端完成
- 无多人交互需求，不需要后端同步
- 部署成本为零（静态托管）
- MVP 阶段快速迭代，后续如需排行榜/云存档再加后端

风险：
- 存档仅在本地，换设备丢失 → 后续可加导出/导入功能
- 无法防作弊 → 单机游戏不需要

## 2. 前端技术栈

**推荐：React + TypeScript + Vite**

| 选项 | 优点 | 缺点 |
|------|------|------|
| **React + TS ✅** | 生态成熟，组件化适合UI密集型，状态管理方案多 | 相对重 |
| Vue | 轻量，上手快 | 生态略小 |
| 原生 JS | 零依赖，极致轻量 | 状态管理和UI更新手写成本高 |

具体选型：
- **构建**：Vite（快速HMR，开箱即用）
- **状态管理**：Zustand（轻量，适合游戏状态树）
- **样式**：CSS Modules 或 Tailwind CSS（纯文字UI，样式简单）
- **无需路由库**：单页面，游戏内用状态切换"场景"

## 3. 数据存储方案

**推荐：localStorage 为主，IndexedDB 备用**

| 方案 | 适用场景 |
|------|----------|
| **localStorage ✅** | 游戏存档（JSON序列化），简单可靠，5-10MB够用 |
| IndexedDB | 如果存档数据量大（大量妖怪/装备记录），升级到这里 |
| 云端 | MVP 不做，后续可加 |

存档策略：
- 每 30 秒自动存档
- 关键操作（升级、收服妖怪）立即存档
- 存档格式：JSON，含版本号用于迁移
- 提供导出/导入存档（Base64编码字符串）

## 4. 离线收益计算方案

**方案：时间差 × 收益速率**

```
离线收益 = min(离线时长, 最大离线时长) × 每秒收益率 × 离线效率系数
```

实现细节：
- 存档记录 `lastSaveTimestamp`
- 打开游戏时：`offlineDuration = Date.now() - lastSaveTimestamp`
- 最大离线时长上限：24小时（防止数值爆炸）
- 离线效率系数：默认 0.5（离线收益为在线的50%，可通过法宝提升）
- 离线期间不触发随机事件（简化计算）
- 显示"离线报告"弹窗：你离开了X小时，获得了Y修为、Z灵石

## 5. 部署方案

**推荐：GitHub Pages + 自定义域名（可选）**

| 方案 | 成本 | 适用 |
|------|------|------|
| **GitHub Pages ✅** | 免费 | MVP 首选 |
| Vercel | 免费 | 备选，支持更多功能 |
| Cloudflare Pages | 免费 | 备选 |

CI/CD：
- GitHub Actions：push 到 main 自动构建部署
- 构建产物：纯静态文件（HTML/CSS/JS）

PWA 支持：
- Service Worker 缓存，支持离线访问
- manifest.json 支持"添加到主屏幕"
- 竖屏锁定：`"orientation": "portrait"`

## 8. 风险与后续

| 风险 | 缓解措施 |
|------|----------|
| 存档丢失 | 导出/导入功能，后续加云同步 |
| 数值膨胀 | 设计阶段做好大数系统（BigNumber） |
| 竖屏适配 | 从一开始就用竖屏优先设计 |
| 纯文字体验单调 | 用好符号、动画（CSS）、音效（后续） |

---

**结论：纯前端 PWA，React+TS+Vite+Zustand，localStorage 存档，GitHub Pages 部署。零后端成本，快速迭代。**
